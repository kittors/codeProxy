# 安全与隐私规范（前端）

本项目涉及“管理密钥（MANAGEMENT_KEY）”与多种 provider key 的管理与展示。目标是：

- 密钥不泄露（不进日志/不进 UI 明文/不进提交）
- 失败信息可用但不敏感

## 1. 敏感信息范围

至少包含：

- 管理密钥（用于 `Authorization: Bearer ...`）
- 任意 provider 的 API Key / token / cookie
- 可能包含密钥的 URL 参数、header

## 2. 禁止项（强制）

- 禁止把密钥打印到 `console.*`
- 禁止把密钥放进 toast/错误信息
- 禁止把密钥写入前端环境变量（会被打包）
- 禁止把包含密钥的示例写进文档/README（使用占位符）

## 3. 本地存储规则

项目存在“记住登录”的能力（localStorage snapshot）：

- 若用户选择不记住：必须清理 snapshot
- 展示层避免把密钥明文渲染（输入框可切换显示，但默认应隐藏）

补充：

- 默认状态下“显示密钥”应为关闭；切换为明文仅用于用户确认输入
- 如果未来增加“复制密钥/复制配置”等按钮，必须明确提示风险，并避免默认复制敏感字段

## 4. 错误信息脱敏策略（建议）

当错误 payload 或文本可能包含敏感信息时：

- 优先展示高层文案（“保存失败，请检查配置”）
- 必要时截断长文本（例如最多 200 字）并避免包含 token 模式

建议脱敏策略（可选）：

- 形如 `sk-...`、`Bearer ...`、`api-key:` 等模式出现时，直接替换为 `***`
- URL 带 query 的情况下，优先移除 query 再展示

## 5. OAuth 相关

涉及 OAuth 的页面：

- 避免把 state/code 等参数写入日志
- 成功/失败提示简洁，不展示完整回调 URL

## 6. 前端安全边界提示（认知）

- 管理后台前端属于“可被用户查看的代码”，任何写进前端的密钥都不安全
- 前端能做的主要是：减少泄露面、避免误展示、降低操作风险
