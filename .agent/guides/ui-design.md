# UI/UX 设计与一致性规范

目标：让后台页面“信息密度高但不压迫、视觉一致、交互可预期”。

## 1. 组件优先复用

先查 `src/modules/ui/*`，优先复用：

- `Button`、`Input`、`Modal`、`ConfirmModal`、`Card`、`Tabs`、`ToastProvider`、`Tooltip` 等

新增 UI 组件时：

- API 要小而清晰（避免大量布尔开关）
- className 只用 Tailwind（禁止原生 CSS/内联 style）
- 必须考虑 dark 模式

## 1.1 页面布局基线（建议）

建议保持与现有页面一致的“后台气质”：

- 背景：使用 `PageBackground`（login/app 两种 variant）
- 容器：以 Card 为基本信息承载单元，减少无边界信息堆叠
- 圆角：以 `rounded-2xl`/更大圆角为主，保持柔和一致

## 2. 信息层级与排版

建议：

- 页面标题/主操作在首屏可见
- 次要信息下沉到 Card/折叠区域
- 文案保持短句、动词开头（例如“保存配置”“刷新数据”）

## 2.1 数值与指标展示（强烈建议）

- KPI/统计类数字使用 `tabular-nums`（避免数字宽度变化抖动）
- 大数格式化保持一致（k/m/b 或千分位），不要同页混用两套规则
- loading 时保持容器高度不变（避免布局跳动）

## 3. 颜色与对比

规则：

- 明暗两套都必须可读（不要靠透明度堆叠到看不清）
- 成功/错误/提示态使用语义色（与 Toast 风格保持一致）

## 3.1 图表颜色一致性（建议）

- 优先复用页面已有颜色序列（例如监控页的颜色常量）
- 同一页面内同一维度尽量保持同色（减少用户认知负担）

## 4. 表单体验

- label 必须清楚描述“是什么/作用/示例”
- 对可能误填的项提供 placeholder 或示例（例如 URL）
- 提交按钮在 loading 时：
  - 禁用
  - 文案稳定（避免布局跳动）

## 4.1 空态与错误态（建议）

- 空态：说明“为什么为空”和“下一步怎么做”（例如“暂无数据，点击刷新”）
- 错误态：提供可重试按钮，并避免覆盖上一次成功数据（如适用）

## 5. 可访问性（最低要求）

- 图标按钮必须有 `aria-label`（例如主题切换、关闭）
- 状态提示区域用 `aria-live`（参考 Toast 容器）
- 表单输入尽量通过 `<label>` 包裹或 `htmlFor` 关联
