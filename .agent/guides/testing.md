# 测试与验证策略（当前以手动验证为主）

说明：本仓库目前未引入专用测试框架（如 Vitest/Testing Library）。在不新增依赖的前提下，本项目的“测试”主要由：

- TypeScript 类型检查（`bun run build`）
- 静态检查（`bun run lint`）
- 手动关键路径验证（见 `workflow.md`）

## 1. 什么时候需要引入测试框架

当出现以下任一情况，建议提出“引入测试框架”的变更提议（需要用户同意）：

- 关键纯函数逻辑不断回归（如 usage 统计/归一化、过滤、格式化）
- 页面交互频繁改动导致回归成本上升
- 多人协作需要更强的质量门禁

推荐方向（需要评估并征得用户确认后落地）：

- 单元测试：Vitest
- 组件测试：@testing-library/react

## 2. 当前可执行的验证清单（不新增依赖）

### 2.1 命令级验证

- `bun run format`
- `bun run check`

### 2.2 手动关键路径验证

最少覆盖：

- 登录（成功/失败/密钥为空）
- 鉴权守卫（未登录访问受保护路由）
- 401 自动退出（触发 unauthorized 事件）
- 主题切换（light/dark，尊重 motion-reduce）
- 监控页刷新（loading 生命周期、错误提示、数值抖动）
