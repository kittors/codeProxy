# API Key 权限配置设计

## 背景

当前 API Key 的渠道分组、精确渠道和模型权限只能在每个 API Key 的编辑弹窗里逐个配置。API Key 数量较多时，用户需要重复进入弹窗，比较和批量调整都不方便；编辑弹窗也因此显得过重。

## 菜单命名

新菜单命名为 **权限配置**。这个名称比“API Key 权限配置”更短，放在现有侧边栏中易读；页面内容和标题再明确说明它用于 API Key 权限。

## 方案对比

### 方案 A：在 API Keys 页面增加批量编辑工具栏

- 优点：改动较少，不增加路由。
- 缺点：仍然和密钥创建、删除、用量入口混在一起；表格横向已经很宽，继续塞批量权限会让页面更重。

### 方案 B：新增独立「权限配置」页面（推荐）

- 优点：权限成为独立工作台，可以一次选择多个 API Key，并统一设置模型、渠道分组和精确渠道。
- 优点：API Key 编辑弹窗可收窄为名称、Key、限额、系统提示词等基础项。
- 优点：和现有“渠道分组”“模型管理”等菜单风格一致，后续也能扩展权限审计。
- 缺点：需要新增路由、菜单、页面和局部 i18n。

### 方案 C：只增加复制权限按钮

- 优点：实现最小。
- 缺点：仍然需要逐个进入编辑弹窗，不满足“很多 apikey 权限一个个配置很麻烦”的主要痛点。

## 结论

采用方案 B：新增独立菜单 **权限配置**，路由为 `/api-key-permissions`。页面集中管理 API Key 的以下权限字段：

- `allowed-channel-groups`
- `allowed-channels`
- `allowed-models`

API Key 编辑弹窗保留基础字段和限制字段：

- 名称
- API Key
- 日请求限制、总请求配额、并发、RPM、TPM
- 系统提示词

## 页面行为

「权限配置」页面加载 API Key、渠道、渠道分组和模型选项。用户可以勾选一个或多个 API Key，在右侧/下方编辑同一组权限草稿，点击保存后批量更新所选 API Key。

当只选择一个 API Key 时，草稿直接展示该 Key 的当前权限。当选择多个 API Key 时，草稿默认保持空限制，用户保存后用草稿覆盖所选 Key 的权限字段；基础信息和限额不受影响。

精确渠道仍作为高级开关展示。默认推荐只使用渠道分组；开启后可限制到具体渠道，并沿用现有“渠道分组和具体渠道取交集”的提示。

## 数据流

- 列表读取沿用 `apiKeyEntriesApi.list()`。
- 保存使用 `apiKeyEntriesApi.replace()`，只改所选条目的三类权限字段。
- 渠道、渠道分组和模型选项复用 API Keys 页面现有加载逻辑，抽到共享 hook，避免重复维护。

## 错误处理

- 加载失败时通过 Toast 展示错误。
- 未选择 API Key 保存时展示提示，不发请求。
- 保存失败时保留当前选择和草稿，便于用户重试。

## 测试

- 路由和菜单测试：确认 `权限配置` 导航、标题和 `/api-key-permissions` 路由存在。
- API Keys 页面测试：确认编辑弹窗不再显示权限选择项。
- 权限配置页面测试：确认可选择多个 API Key，并批量保存渠道分组、精确渠道和模型权限。

## 影响范围

- 新增 `src/modules/api-key-permissions/` 页面模块。
- API Key 页面抽出共享加载 hook，并简化弹窗字段。
- 新增路由和侧边栏导航。
- 更新中英文 i18n、`AGENTS.md` 关键路径和 `docs/evolution.md` 演进记录。

## 回滚策略

删除 `/api-key-permissions` 路由、侧边栏导航和新页面模块；将权限字段重新挂回 `ApiKeyFormFields`；移除新增 i18n、测试和文档记录即可回退。
