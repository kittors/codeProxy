# API Key 权限配置设计

## 背景

API Key 的权限、限额和系统提示词如果全部放在单个密钥编辑弹窗中，数量一多就会变成重复劳动。更合适的模型是先维护可复用配置，再让 API Key 选择配置。

## 菜单命名

新菜单命名为 **权限配置**。侧边栏保持简洁，页面标题和表格内容说明它服务于 API Key。

## 结论

新增独立「权限配置」页面，路由为 `/api-key-permissions`。页面不直接批量勾选 API Key，而是维护 `api-key-permission-profiles` 配置列表。

每条配置包含：

- 渠道分组、精确渠道、模型权限
- 每日请求限额、总请求配额、并发请求限制、RPM、TPM
- 系统提示词

API Key 新建/编辑弹窗只保留名称、Key 和「权限配置」下拉。选择配置后，将配置内容复制到该 API Key 的实际字段中，并记录 `permission-profile-id`。未选择配置时默认不限制。

## 行为细节

- `权限配置` 页面使用 `VirtualTable` 展示配置列表。
- 新增/编辑配置通过弹窗完成，数值留空或填 0 都表示不限制。
- 编辑已有配置后，所有绑定该配置的 API Key 会同步更新为新配置内容。
- 删除配置时，已绑定的 API Key 保留已复制的字段，并移除绑定标记，作为自定义配置保留。
- 如果旧 API Key 已有自定义权限但没有绑定配置，编辑弹窗显示「当前自定义配置（保留）」，保存基础信息不会清空原字段。

## 数据流

- 配置列表从 `config.yaml` 的 `api-key-permission-profiles` 读取和写回。
- API Key 仍通过 `apiKeyEntriesApi` 保存实际生效字段，保证后端执行逻辑不依赖前端模板。
- 渠道、渠道分组和模型选项继续复用 `useApiKeyPermissionOptions`。

## 测试

- API Key 弹窗测试：确认限额、权限、系统提示词字段不再直接出现；选择配置后创建的 Key 写入配置字段。
- 权限配置页面测试：确认页面展示配置列表、可新增配置，并写回 YAML。
- 既有认证文件缓存测试：补齐最新缓存字段期望，恢复全量测试基线。

## 回滚策略

移除 `/api-key-permissions` 路由、侧边栏导航、配置列表页面和 `api-key-permission-profiles` 读写封装；将限额、系统提示词和权限选择器重新放回 API Key 编辑弹窗。
