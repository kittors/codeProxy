# 开发工作流规范（渐进式）

本规范用于把“写代码”变成“可交付的工程改动”，重点是：**最小改动、可验证、可回滚（非破坏性）**。

## 1. 任务开发流程（通用）

```
┌──────────────────────────────────────────────────────────┐
│                       开发工作流                           │
├──────────────────────────────────────────────────────────┤
│  1. 理解需求                                                │
│     └─→ 明确页面/模块/接口范围                               │
│     └─→ 明确成功标准（UI/行为/性能/兼容）                    │
│                                                           │
│  2. 设计方案                                                │
│     └─→ 确定目录位置（参考 architecture.md）                 │
│     └─→ 是否涉及接口变更（参考 http-api.md）                 │
│     └─→ 评估改动范围与风险                                  │
│                                                           │
│  3. 编码实现                                                │
│     └─→ 遵循 Tailwind/无原生CSS（参考 tailwind.md）          │
│     └─→ 异步 loading 与错误提示绑定生命周期                  │
│     └─→ 避免 remount/布局抖动（tabular-nums/min-w-*）        │
│                                                           │
│  4. 自测验证                                                │
│     └─→ 关键路径手动验证（登录/守卫/页面加载/错误态）         │
│     └─→ bun run check                                       │
│                                                           │
│  5. 交付与说明                                              │
│     └─→ 总结改动点、风险点、验证方式                         │
└──────────────────────────────────────────────────────────┘
```

## 2. 功能开发 Checklist（推荐顺序）

### Phase 1：边界与设计
- [ ] 明确要改的页面与入口路由（`src/app/AppRouter.tsx`）
- [ ] 明确是否需要新增/调整 API（如需要，先读 `http-api.md`）
- [ ] 明确复用组件是否已有（先看 `src/modules/ui/*`）

### Phase 2：实现
- [ ] UI：只使用 Tailwind class，不写原生 CSS、不写内联 style
- [ ] 异步：loading 与真实生命周期绑定，失败有中文提示
- [ ] 性能：避免用 `key` 触发整块 remount；必要时用 `useTransition`

### Phase 3：自测与质量门禁
- [ ] 运行 `bun run format`
- [ ] 运行 `bun run check`
- [ ] 手动自测关键路径（见第 4 节）

## 3. Bug 修复流程（严格最小化）

1. **重现**：描述最短重现路径（页面/按钮/输入/响应）
2. **定位**：找出根因（API 响应结构、状态管理、渲染条件、边界值）
3. **修复**：最小改动修根因，不顺手重构不相关代码
4. **验证**：回归关键路径，确认不引入新问题

原则：

- 先保证“用户可用”，再谈“更优雅”
- 不把“顺手重构”混进 Bugfix diff（可另开任务）

## 4. 手动自测清单（本项目必测）

- 登录页：
  - API 地址规范化是否正确显示“连接地址”
  - 管理密钥为空时提示是否清晰
  - 登录失败提示是否不泄露敏感信息
- 鉴权：
  - 未登录访问受保护路由是否跳转登录
  - 401 后是否自动退出登录态（`unauthorized` 事件）
- 主题：
  - 亮/暗切换是否只通过 `html.dark` 生效
  - 切换是否无明显闪烁（动效尊重 motion-reduce）
- 监控页：
  - 刷新 loading 是否准确开始/结束
  - 数字展示是否 `tabular-nums`，是否存在抖动

## 5. 代码审查自查清单（提交前）

### 工程与安全
- [ ] 没有新增依赖（如有，已得到用户确认）
- [ ] 没有修改 `dist/`、`node_modules/`
- [ ] 没有把密钥/Token 写进代码、日志、错误提示

### 架构与一致性
- [ ] API 调用只通过 `src/lib/http/apis.ts`
- [ ] 401 处理没有在页面里重复实现
- [ ] 新增组件复用现有 `src/modules/ui/*`（或给出原因）

### UI/体验
- [ ] 没有原生 CSS / 内联 style
- [ ] loading 与真实异步生命周期绑定
- [ ] 数值 `tabular-nums`，必要时使用 `min-w-*` 降低布局抖动

### 可观测性与验证
- [ ] 如新增/修改日志输出，已读 `logging.md` 并确保脱敏
- [ ] 如改动影响范围较大，已按 `testing.md` 扩展手动验证清单
