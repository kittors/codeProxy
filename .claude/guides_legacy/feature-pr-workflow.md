# 功能开发与 PR 工作流（在本仓库语境下）

说明：

- 是否使用 PR 取决于你的团队流程；本文件提供“如何把一个功能做成可 Review 的变更集”
- **Git 安全规则仍然强制生效**：任何 `fetch/pull/push` 或破坏性操作必须先征得用户同意（见 `git.md` 与 `AGENTS.md`）

## 1. 功能开发的提交组织策略

目标：让 Review 能快速理解“做了什么/为什么这样做/风险在哪里”。

建议把一个功能拆成 2～4 个逻辑提交（视复杂度而定）：

1. `chore/docs`：补充/更新规范与文档（若需要）
2. `refactor`：必要的抽取与铺垫（最小化，不夹带功能）
3. `feat`：核心功能改动
4. `fix`：修复评审过程中发现的问题

避免：

- 一个提交同时包含重构 + 功能 + 格式化全仓库（难以 Review）
- “顺手整理”导致 diff 巨大

## 2. Code Review 自查清单（提交前）

### 2.1 行为正确性
- [ ] 路由入口与守卫逻辑正确（`AppRouter`/`ProtectedRoute`）
- [ ] loading 与异步生命周期绑定
- [ ] 错误提示中文且可行动
- [ ] 401 行为一致（由 ApiClient 事件驱动）

### 2.2 架构一致性
- [ ] API 调用只通过 `apis.ts`
- [ ] 数据归一化在 `apis.ts` 完成，页面不重复解析
- [ ] UI 复用 `modules/ui` 组件（或说明原因）

### 2.3 样式合规
- [ ] 无原生 CSS/内联 style
- [ ] dark: 变体覆盖到新增 UI
- [ ] 数值展示 `tabular-nums`，避免布局抖动

### 2.4 安全与隐私
- [ ] 没有输出/展示任何密钥或 token
- [ ] 示例文本均为占位符

### 2.5 质量门禁
- [ ] `bun run format`
- [ ] `bun run check`
- [ ] 手动自测关键路径（见 `workflow.md`）

## 3. 变更说明模板（给 Review/交付用）

建议在交付说明中包含：

- 需求目标与范围
- 改动点（按文件/模块）
- 风险点与兼容策略
- 验证方式（命令 + 手动路径）
