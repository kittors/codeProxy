# 规范演进（渐进式维护）

本项目的规范应当“可演进”，但必须避免把规范变成负担。

## 1. 什么时候需要更新规范

- 出现反复踩坑的问题（例如 API 解析、错误提示、样式约束被误用）
- 项目引入新工具链/新模块边界
- 新成员加入导致一致性下降

## 2. 变更流程（建议）

1. 提议：说明要解决的问题与现状痛点
2. 最小变更：优先修改对应 `.claude/guides/*.md`
3. 同步入口：如属于“强制规则”，同时更新 `AGENTS.md`
4. 验证：用一个真实任务验证规范可落地

## 3. 兼容策略

- 新规范尽量向后兼容，避免一次性要求全量重构
- 若必须破坏性变更（例如目录调整），需要明确迁移步骤与回滚方案
