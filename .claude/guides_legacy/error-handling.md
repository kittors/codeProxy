# 错误处理规范（用户可理解 + 行为一致）

目标：

- 错误提示“可行动”（用户知道怎么修）
- 行为一致（401/超时/解析失败不会各自一套）
- 不泄露敏感信息（管理密钥等）

## 1. 错误来源分类

1) **用户输入错误**（如管理密钥为空）

- 直接在页面层阻止提交，并 toast 提示

2) **网络/超时/取消**

- `ApiClient` 使用 `AbortController` + timeout
- 页面层应展示“请求失败/超时”并提供重试入口（按钮/刷新）

3) **HTTP 非 2xx**

- `ApiClient` 会抛出 `Error(message)`
- message 可能来自服务端 `error`/`message` 字段或文本

4) **数据结构不符合预期**

- 优先在 `apis.ts` 做兼容与归一化
- 页面层尽量只处理“已经归一化”的结构

## 2. 401（未授权）处理（必须一致）

规则：

- `ApiClient` 发现 401：派发 `unauthorized` 事件
- `AuthProvider` 监听事件：统一清理会话、退出登录态
- 页面 **禁止** 自行对 401 做跳转或清理（避免重复与竞态）

## 3. 页面层标准处理模板（原则）

你在页面里处理异步时必须满足：

- `setLoading(true)` 在请求前
- `finally` 中 `setLoading(false)`
- `catch` 中把错误转换为中文可理解文案

并且：

- 不把密钥拼进错误信息
- 不输出服务端返回的完整敏感内容（若不确定，做截断/脱敏）

补充建议：

- 对“可重试”的错误（网络/超时）提供重试按钮或刷新入口
- 对“不可重试”的错误（鉴权失败/参数错误）提示用户下一步（检查地址/检查密钥/联系管理员）

## 4. toast 与页面错误的取舍

建议：

- 表单提交失败：toast（同时保持表单可编辑）
- 页面核心数据加载失败：页面内错误态 + 提供重试按钮（可辅以 toast）
- 非关键提示：info toast（时间短）

经验法则：

- 影响“整页可用性”的错误：页面内错误态优先
- 影响“单次操作”的错误：toast 优先
- 同一错误不要 toast + 页面错误重复刷屏（二选一或弱化其中一个）

## 5. 错误文案规范（中文）

文案应包含：

- 发生了什么：登录失败/数据获取失败/保存失败
- 用户能做什么：检查地址/检查密钥/稍后重试

禁止：

- 直接展示“未知错误：Error: …”这类开发者文本
- 直接把后端堆栈或整段 HTML/JSON 丢给用户（必要时截断）

## 6. 与 ApiClient 的配合点

当你需要定位错误来源时，先确认：

- 是否是超时导致（默认 30s）
- 是否是 401 触发（会自动退出登录态）
- 是否是响应体解析失败（ApiClient 会尽力解析 JSON，否则回退文本）

如果发现 ApiClient 的错误解析规则不足以覆盖某类后端返回，请优先在 `ApiClient.request()` 内补强解析逻辑（但必须确保不影响现有行为）。
